Caching, from LRU to LLMs

the classic patterns, the reading list, and where it meets ML

Caching is one of those topics where the fundamentals were mostly settled in the 1970s and yet every generation of infrastructure invents a new twist on them. This guide walks the classic patterns first, points at the engineering-blog posts worth reading, then follows the thread into modern ML systems — feature stores, embedding caches, and the semantic and KV caches that make LLM serving affordable.

1. Why we cache

Three reasons, in order of how often they matter:

  1. Latency. Memory is roughly 100× faster than SSD, which is 100× faster than a network round-trip, which is 100× faster than a slow origin request. A cache turns "slow" work into "fast" work by keeping recent results near where they're needed.
  2. Load. A cache absorbs traffic that would otherwise hammer the origin. If 90% of reads hit the cache, the database sees 10% of the load.
  3. Cost. Especially with LLMs: every cache hit is a call you don't pay a per-token bill for.
The mental model: a cache is a bet that recent work will happen again soon. When that bet pays off it's called temporal locality. Every caching decision — what to keep, what to evict, how long to trust it — is a variation on tuning that bet.

Big services use several of these at once. A request might hit a CDN, then an in-process cache on the app server, then a Redis cluster, and only fall through to Postgres on a miss at every level.

3. Read patterns

Cache-aside (aka lazy loading)

The application does the work. Check cache; on miss, fetch from DB and populate the cache.

def get_user(user_id):
    user = cache.get(user_id)
    if user is not None:
        return user           # HIT
    user = db.query("select ...", user_id)   # MISS
    cache.set(user_id, user, ttl=300)
    return user

Dominant in practice because it's simple and only caches things that are actually asked for. The tradeoff: every miss is a round-trip to the cache and a round-trip to the DB.

Read-through

Same behavior, but the cache library is in charge of the fallback. The application only ever calls cache.get(). Nicer abstraction, less flexible when the "fetch from origin" logic is complicated.

4. Write patterns

PatternHow it worksTradeoff
Write-through Every write goes to cache and DB, synchronously. Consistent, but every write pays two hops.
Write-back (write-behind) Write to cache only; a background flush persists to DB later. Fast writes, but a cache crash before flush loses data.
Write-around Write straight to DB; do not populate the cache on write. Good when written data is rarely read soon after.

Real systems mix these. A common combo is cache-aside for reads plus write-through for updates, so the cache never returns a value it just watched the DB accept.

5. Eviction policies — and an LRU playground

A cache has finite capacity, so on every write beyond capacity it must evict something. The eviction policy is the rule for choosing what to throw out.

  • LRU (Least Recently Used) — evict the item touched furthest in the past. The default choice; matches temporal locality.
  • LFU (Least Frequently Used) — evict the item with the fewest hits. Better when access frequency is more stable than recency.
  • FIFO — evict oldest inserted. Cheap; usually worse than LRU.
  • TTL-based — every entry has an expiry; expired entries are lazily or actively removed.
  • ARC / W-TinyLFU — adaptive hybrids that mix recency and frequency. Caffeine and modern in-process caches use W-TinyLFU.

6. Cache stampede

Imagine a popular key expires at exactly 12:00:00. In the next millisecond a thousand requests all miss the cache, all query the database, and all write the same value back. Your DB just took a 1000× spike from a single expired key. This is a cache stampede (also called the thundering herd or dog-pile problem).

Standard defenses:

  • Request coalescing / singleflight. Only one request per key is allowed to fetch from the origin; the others wait for its result. Go's golang.org/x/sync/singleflight is the reference implementation of this idea.
  • Probabilistic early expiration. Refresh a key before it expires, with a probability that grows as the TTL nears zero. See the XFetch algorithm.
  • Jittered TTLs. Add random noise so a batch of keys populated together doesn't expire together.
  • Stale-while-revalidate. Serve the expired value while a single background task refreshes it. Standard in CDNs and HTTP.

7. Invalidation, TTLs, and consistency

The old joke — "there are only two hard things in computer science: cache invalidation and naming things" — is a joke because it's true. If you never invalidate you serve stale data. If you invalidate on every write you've reinvented the database, slower.

Common strategies, roughly in order of increasing complexity:

  1. TTL only. Every entry lives for N seconds. Simple, tolerates staleness bounded by N. Usually the right default.
  2. Write-through invalidation. The code that updates the DB also deletes (or overwrites) the corresponding cache key.
  3. Event-driven invalidation. The DB emits change events (Debezium, DynamoDB Streams, Postgres logical replication); a consumer invalidates keys.
  4. Versioned keys. Include a version number in the cache key (user:42:v7). "Invalidate" by bumping the version — old entries just expire naturally.
A subtle bug worth remembering: if you update the DB and then delete the cache key, there's a race where another reader can miss, read the old value from the DB (before your commit is visible on a replica), and repopulate the cache with stale data. The fix is either "delete-then-update-then-delete-again" (the double-delete pattern) or stronger read consistency.

8. Consistent hashing (briefly)

When your cache is a cluster of nodes, how do you decide which node holds which key? Naive hash(key) % N rebalances almost every key when you add or remove one node — a disaster during a scaling event.

Consistent hashing maps both keys and nodes onto a ring (typically 232 positions). A key belongs to the next node clockwise on the ring. Adding or removing a node only moves the keys in that node's arc — on average, K/N of them. This is how Memcached clients, Cassandra, DynamoDB, and every serious cache cluster do sharding. Worth reading the original Karger et al. paper once, then never having to think about it again.

9. Engineering blog reading list

The theory is short. The interesting stuff is how big companies actually made it work. Read these roughly in order — the first few build the intuition, the later ones stretch it.

  • Facebook · NSDI '13 paper
    Scaling Memcache at Facebook

    Still the canonical read. Covers stampede protection ("leases"), regional pools, and cache consistency across data centers. Referenced by every serious caching post since.

  • CodeAhoy
    Caching Strategies and How to Choose the Right One

    Clean tour of cache-aside, read-through, write-through, write-back, refresh-ahead. Good after the Facebook paper for putting names on patterns.

  • Netflix
    EVCache: Caching for a Global Netflix

    Multi-region distributed cache built on Memcached. Great case study in what happens when your users are everywhere and eventual consistency is the only realistic option.

  • Uber
    How Uber Serves Over 40 Million Reads Per Second Using an Integrated Cache

    CacheFront, an in-Docstore cache. Shows what it looks like when the cache is designed as part of the database rather than bolted on.

  • Shopify
    The Story of IdentityCache

    A read-through cache layered on ActiveRecord/MySQL. Very practical — you can feel the production scars in the design decisions.

  • Discord
    How Discord Stores Billions of Messages

    Not strictly a caching post, but the access-pattern analysis (why they picked Cassandra, how they warm caches, hot-partition mitigation) teaches you how to think about caches.

  • Cloudflare
    How We Built Pingora

    The proxy that fronts Cloudflare's cache. Search their blog for "cache" — there are dozens of posts on edge caching, tiered caches, and cache-key normalization.

  • Instagram Engineering
    Instagram Engineering — Django cache posts

    Older but foundational. The "how we sharded Postgres and used memcached" era of scaling writeups.

10. Caching for ML systems

Machine learning turned out to be a giant caching problem in disguise. Every layer of a modern ML system has a cache tuned to a specific access pattern:

Feature stores — the online cache for ML

A model at inference time needs the same features that were used at training time. Recomputing them per-request is too slow. So features are computed in batch, written into an online store optimized for point lookups by entity id, and cached aggressively. That's a feature store.

  • Uber
    Meet Michelangelo: Uber's Machine Learning Platform

    The post that popularized the term "feature store". Read it for the split between offline (Hive) and online (Cassandra) feature retrieval.

  • DoorDash
    Building a Gigascale ML Feature Store with Redis

    Very concrete: Redis Cluster, cache warming, hot-key handling. Reads like the ByteByteGo article but the "value" is an ML feature vector.

  • LinkedIn
    Open-sourcing Venice: LinkedIn's Derived Data Platform

    Venice is the online serving side of their ML pipelines — a bulk-write, point-read store built specifically for derived (model-output) data.

  • Open source
    Feast · Tecton blog

    The open-source and commercial feature stores. Their docs and blogs are essentially a tour of every caching problem you've read about, framed for ML.

Embedding & vector caches

Anywhere you have embeddings — recommendation towers, RAG retrievers, ad ranking — you're caching high-dimensional vectors. Vector databases (Pinecone, Weaviate, Milvus, pgvector) all have hot-key caches sitting in front of their ANN indexes. The hard part is that "the same query" means "close in vector space", not "byte-equal" — which brings us to…

11. Semantic cache playground

A semantic cache answers a query if any previously-seen query was semantically similar enough. This is how you can cache LLM responses even when users phrase the same question a dozen ways.

Real semantic caches encode each query as a vector (using an embedding model), then compare with cosine similarity. 12. LLM KV cache and prompt caching

KV cache (inference-time, inside the model)

Transformers compute an attention "key" and "value" for every previous token when generating the next one. Recomputing them per generated token would be quadratic. Instead, the model caches them — the KV cache — and grows the cache by one entry per generated token. This cache dominates GPU memory in long-context serving.

Two papers/systems worth knowing:

  • vLLM & PagedAttention — borrows OS-style paging to store the KV cache in fixed-size blocks, which massively raises effective batch size.
  • Character.AI — Optimizing Inference — production war stories on KV cache reuse across turns.

Prompt caching (across API calls)

A newer layer: if your requests share a long prefix — a system prompt, a big context document — the provider will cache the prefix's KV tensors and skip re-encoding it on subsequent calls. You pay a discounted per-token rate for the cached portion.

  • Anthropic — Prompt caching (explicit, opt-in via cache-control breakpoints)
  • OpenAI — API prompt caching (automatic for prefixes over a threshold)
  • GPTCache — open-source semantic cache library, if you want to see the pattern from section 11 built out as production code.
Design lens: classic caching asks "was this exact key requested before?" Semantic caching asks "was anything close enough requested before?" KV / prompt caching asks "does this request share a prefix with an earlier one?" All three are the same bet on temporal locality; the definition of "same" is what varies.
